noexcept + throw

发表于 2023-08-12 更新于 2024-03-06 分类于 C++ Language

noexcept 有几个用法：

  1. 在编译期返回一个常量布尔值，评估其表达式是否会抛出异常。
  2. 用于标志一个函数是否能抛出异常，需要一个编译期布尔常量作为参数。
  3. 标志函数不会抛出异常。相当于 noexcept(true)。

C++17 已经禁用 throw 的显式异常声明。（被称为 Dynamic exception specification）

尝试用 std::type_index 代替 dynamic_cast？

可以用 std::type_index 来唯一标记数据的类型，该结构占用 8 个字节，且能够作为 key 放入容器中。std::type_index 能够判断一个类是不是目标类型，而 dynamic_cast 能够判断一个类是不是目标类型或其子类，所以前者的判断更加精确。

在 bustub 的例子中，由于 TrieNodeWithValue<T> 在不同的 T 之间没有继承关系，所以 dynamic_cast 成功后可以确定是相同的类型，能满足功能的要求。

💡 理论上，如果 B 继承于 A，那么从一个包含 B 对象的结点获取 A 类型的指针也应该是可能的。但是 TrieNodeWithValue<B> 并不继承于 TrieNodeWithValue<A>，这使得涉及向上转型的 Get 操作无法在给定的代码中实现，从而本质上是完成了“精确获取”。

💡 std::type_index 需要结合 typeid 来使用，std::type_index 和 typeid 都有唯一性，但是 typeid(expr_or_type).name() 不一定有。详见 cppreference。

能观察到：

  1. 查虚表在不打开优化选项时比普通函数需要多访问两次内存。
  2. 来自不同基类的虚函数会在不同的虚表段里去查，将上面的 p->f1(); 改成 p->f2(); 重新编译就能看出来。

  5. 对象中存储的虚函数表指针其实对应于虚表段 vtable_prefix 的 origin 域的地址。虚函数表至少有一个函数，否则这个表项可以被消除。

对 {} 初始化的理解

可以对基本类型和聚合类零填充。

类型推导分类

见 01 Function Templates。下面讨论的是其中的第 3、4 类。

（嵌套）类型推导

花括号的规则（大概？）是：

  1. 先看对应位置可能出现什么类型，再尝试在花括号前添加这个类型。
  2. B a{3}; 可以看成 B a = {3};
  3. 花括号推导在面临多种推导可能时，会尽可能选择外层和所需类型最接近的一个（例子见下面代码中的 B g2 {{5}}; ）。注意花括号推导并不是函数重载选择，不会因为有多个推导可能而让编译器抱怨 ambiguous。

在 std::vector<int> v {1, 2, 3}; 的情况下，{1, 2, 3} 被认为是 std::initializer_list<int>，所以对 std::vector<int> 初始化。std::vector<int> v {{1, 2, 3}}; 也是等效的，这是其他类型正常初始化的方式（但对容器来说比较冗余）。

花括号本身可以被解释成std::initializer_list<T>（解释不计入类型隐式转换次数）。这是由编译器支持的花括号类型推导的额外规则，因而 auto a = {1, 2}; 虽无规则可用，还是能够被推导成 std::initializer_list<int>。

从 f 不能构造成功但是 g2 可以这一点能看出：花括号初始化并不等于直接调用构造函数，尽管花括号在推导完后最终会使用构造函数构造出目的类型。
