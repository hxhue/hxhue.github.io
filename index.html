向下转换是可能出现溢出的转换。检查是否为向下转换：

其中 To{std::declval<From>()} 这一步如果用在具体的类型上（比如说 int 和 unsigned），编译器是只给警告，不会报错。如果能够在编译期推导出来数值，则会根据数值来选择通过或报错。比如 char{'a'}; 是正常编译的，但是 char{300}; 是会报错的。

向下转换溢出的判断标准是真实的表数范围：

  1. 无符号数和对应的有符号数互转都算是溢出。
  2. 范围较小的无符号数转向范围较大的有符号数是不溢出的。
  3. 任何有符号数转向无符号数都会溢出，因为无符号数不能表示负数。

溢出的定义情况

算术运算溢出只有在 unsigned 类型上有定义，在浮点数上无定义，在 signed 整数上是 implementation-defined（C++17）或模除定义（C++20，同 unsigned）。

因此，虽然可能会溢出，大范围的数字转向任意范围的无符号数都有良好定义。大范围的数字转向小范围的有符号数，在溢出的时候仅在 C++20 以上有良好定义（否则是实现定义）。

(CppCon 2023) Lock-free Atomic Shared Pointers Without a Split Reference Count

发表于 2024-03-05 更新于 2024-03-10 分类于 cppcon
本文字数： 410 阅读时长 ≈ 1 分钟

https://youtu.be/lNPZV9Iqo3U?si=lS1hf2ND4SS-6ELB by Daniel Anderson

std::shared_ptr 的原子引用计数是线程安全的，但是：

  1. 对它指向的对象的访问不是线程安全的
  2. 对它本身的读写不是线程安全的

这个 lecture 主要考虑第 2 点。在有一个全局或者类似全局的共享指针时，需要注意这个。

首先不能使用 std::atomic<std::shared_ptr>，因为这个类型不是无锁的，有造成阻塞的可能。

然后可以考虑用一个单独的原子变量来计数（这样就会同时有一个 std::atomic<int> 和一个 std::shared_ptr），确保 store 操作和 load 操作的安全性（不会 load 到一个引用计数已经归零然后被删掉的 std::shared_ptr 或者说控制块；一旦控制块被 delete，其他线程在控制块上的查询操作即对引用计数的原子读就会出现非法内存访问。）

有些库甚至更进一步地实现了 split reference counted shared pointer，也就是把上一段的机制集成在了 std::shared_ptr 的内部，这比在外围做并发保护效率会高一些。

然后 lecture 的重点是介绍 hazard pointer。使用 hazard pointer 保护对象时，控制块引用计数归零可以删掉对象，但不是马上 delete 掉这个控制块，而是使用 retire 方法，使得其他线程能够继续查询控制块。在没有线程查询控制块时再删除 retired control blocks，同样是把复杂度从用户移向了库的实现者。C++26 有可能会加入这个头文件。

常量内存

常量内存是在全局区域声明的。如果漏掉了 __constant__ 关键字，就会将其定义在全局内存区域，尽管存储方式、分配的时机和用 cudaMalloc 申请的内存有一些差异。

CUDA 线程对常量内存是只读的，也就是只有 host 能操作常量内存。通过将反复读取的数据移动到常量内存区域而不是全局内存，可以加速。但是要注意常量内存的大小非常有限（）。书中的例子只是对 20 个球体做光线追踪。

常量内存为什么快：

  1. 有 constant cache。
  2. 对常量内存的读取可以被广播到临近的线程。节省读取次数。

什么叫做临近线程？GPU 中线程执行是按照 warp 分组的。如果同一组的都需要同一个 constant memory 中的数据，那么 GPU 只会产生一个访问请求，这样会很节省带宽。（需要全局内存时难道不能用这个方法节省带宽吗？）

第 3 章 CUDA 源文件

用 nvcc 编译时不需要为 cuda 内置函数额外包含头文件。这些头文件是在 host 端才需要的。

https://stackoverflow.com/questions/6302695/difference-between-cuda-h-cuda-runtime-h-cuda-runtime-api-h

In very broad terms:

  • cuda.h defines the public host functions and types for the CUDA driver API.
  • cuda_runtime_api.h defines the public host functions and types for the CUDA runtime API
  • cuda_runtime.h defines everything cuda_runtime_api.h does, as well as built-in type definitions and function overlays for the CUDA language extensions and device intrinsic functions.

If you were writing host code to be compiled with the host compiler which includes API calls, you would include either cuda.h or cuda_runtime_api.h. If you needed other CUDA language built-ins, like types, and were using the runtime API and compiling with the host compiler, you would include cuda_runtime.h. If you are writing code which will be compiled using nvcc, it is all irrelevant, because nvcc takes care of inclusion of all the required headers automatically without programmer intervention.
